```c
int p[2], n;
pipe(p);
if (!fork()) {
    // Παιδί
    char buffer[32] = {'\0'};
    write(p[1], "World", 5);
    close(p[1]);
    n = read(p[0], buffer, 100);
    printf("%d %s\n", n, buffer);
} else {
    // Γονιός
    char buffer[32] = {'\0'};
    write(p[1], "Hello", 5);
    close(p[1]);
    n = read(p[0], buffer, 100);
    printf("%d %s\n", n, buffer);
}
close(p[0]);
```

### Κύρια σημεία:

Το pipe(p) δημιουργεί ένα κοινό pipe με δύο άκρα: p[0] για ανάγνωση και p[1] για εγγραφή.

Το fork() δημιουργεί παιδί. Και τα δύο (γονιός και παιδί) κληρονομούν και τα δύο άκρα του pipe.

ΚΑΙ ο γονιός ΚΑΙ το παιδί γράφουν πριν διαβάσουν, και μετά κάνουν read από το ίδιο άκρο (p[0]).

Το close(p[1]) γίνεται ΠΡΙΝ την read, και αυτό προκαλεί EOF όταν όλες οι διεργασίες κλείσουν το p[1].

### Πιθανές Συμπεριφορές:

Τα δεδομένα που γράφονται στο pipe συσσωρεύονται, δηλαδή αν γραφεί πρώτα "World" και 
μετά "Hello", τότε το read() μπορεί να επιστρέψει "WorldHello" ή "HelloWorld", ανάλογα με το ποιος γράψει πρώτος.

Δεν υπάρχει συγχρονισμός μεταξύ του ποιος γράφει ή διαβάζει πρώτος, άρα η σειρά εκτέλεσης είναι μη καθορισμένη.


### Ανάλυση Επιλογών:
✅ A. Μπορεί να εκτυπωθεί 10 HelloWorld δύο φορές (μία από τον γονιό και μία από το παιδί).

✅ B. Μπορεί να εκτυπωθεί 10 WorldHello αποκλειστικά από τον γονιό ή αποκλειστικά από το παιδί.

Πιθανό: Αν ο ένας κάνει και τα δύο write και read πρώτος, μπορεί να πάρει ολόκληρο το περιεχόμενο του pipe.

Απίθανο: Ο καθένας κάνει write στο p[1] και μετά read από το ίδιο p[0]. Άρα δεν εγγυάται να διαβάσει αυτά που έγραψε ο άλλος.

✅ G. Μπορεί να εκτυπωθεί 0 (και κενή συμβολοσειρά) είτε από τον γονιό είτε από το παιδί.

```c
#include <stdio.h>
#include <unistd.h> // Για fork() και execlp()

int main (int argc, char *argv[]) {
    int x = 5;

    fork();
    x++;
    if (!fork()) {
        x--;
        printf("%d\n", x);
        execlp("./prog", "prog", NULL); // Υποθέτουμε επιτυχή εκτέλεση του './prog'
        x++; // Αυτό δεν θα εκτελεστεί ποτέ μετά το execlp
    }
    x++;
    printf("%d\n", x);
    return 0;
}
```

## Υπόθεση

Υποτίθεται ότι η εκτέλεση του προγράμματος `./prog` (μέσω `execlp`) είναι **επιτυχής**. Αυτό σημαίνει ότι οι διεργασίες που καλούν `execlp` **αντικαθίστανται** από το `./prog` και **δεν συνεχίζουν** την εκτέλεση του αρχικού κώδικα μετά την κλήση αυτή.

## Βήμα προς Βήμα Ανάλυση

1.  **Αρχικοποίηση:**

      * `int x = 5;`

2.  **Πρώτη `fork()`:**

      * `fork();`
      * Δημιουργούνται **δύο** διεργασίες:
          * **Διεργασία 1 (Αρχικός Γονέας):** `x = 5`
          * **Διεργασία 2 (Πρώτο Παιδί):** `x = 5`

3.  **`x++;`:**

      * Και οι δύο ενεργές διεργασίες αυξάνουν την τιμή του `x`.
          * Διεργασία 1: `x = 6`
          * Διεργασία 2: `x = 6`

4.  **Δεύτερη `fork()` εντός `if (!fork())`:**

      * Η `fork()` εκτελείται από τη Διεργασία 1 και τη Διεργασία 2.
      * **Μόνο** τα **νέα παιδιά** που δημιουργούνται από αυτή τη `fork()` θα εισέλθουν στο `if` μπλοκ (καθώς `fork()` επιστρέφει `0` στο παιδί).
      * Δημιουργούνται **δύο νέα παιδιά**:
          * **Διεργασία 3 (Παιδί της Διεργασίας 1):** `x = 6` (εισέρχεται στο `if`)
          * **Διεργασία 4 (Παιδί της Διεργασίας 2):** `x = 6` (εισέρχεται στο `if`)
      * Οι Διεργασίες 1 και 2 **δεν** εισέρχονται στο `if` μπλοκ.

5.  **Εκτέλεση εντός `if` μπλοκ (Διεργασίες 3 και 4):**

      * `x--;` : `x` γίνεται `5`.
      * `printf("%d\n", x);` : **Εκτυπώνεται το `5` (δύο φορές)**.
      * `execlp("./prog", "prog", NULL);` : Αυτή η κλήση **αντικαθιστά** τις διεργασίες 3 και 4 με το πρόγραμμα `./prog`. Ο κώδικας που ακολουθεί (π.χ. το `x++`) **δεν εκτελείται ποτέ** από αυτές τις διεργασίες.

6.  **Εκτέλεση μετά το `if` μπλοκ (Διεργασίες 1 και 2):**

      * Αυτές οι διεργασίες παρέμειναν με `x = 6`.
      * `x++;` : `x` γίνεται `7`.
      * `printf("%d\n", x);` : **Εκτυπώνεται το `7` (δύο φορές)**.
      * `return 0;` : Οι διεργασίες τερματίζουν.

  * **Το 6 θα εκτυπωθεί:** 0 φορές.

## Question 4

This document explains the behavior of a given C code snippet that manipulates two files, `file1` and `file2`.

## Scenario Description

We are given two initial conditions for our files:

  * `file1` initially contains the text: `ONE-TWO-THREE-`
  * `file2` initially contains the text: `123456789`

We are asked to determine the final content of `file1` after the execution of the provided C code.
**Assumption:** All system calls (like `open`, `read`, `write`, `close`) are successful.

## The C Code

```c
int fd1, fd2, x;
char s[20];

// Open file1 for write-only access. This truncates the file if it exists.
fd1 = open(file1, O_WRONLY);

// Open file2 for read-only access.
fd2 = open(file2, O_RDONLY);

// Read up to 20 bytes from file2 into buffer 's'.
x = read(fd2, s, 20);

// Write 'x' bytes from buffer 's' into file1.
write(fd1, s, x);

// Close both file descriptors.
close(fd1);
close(fd2);
```

## Code Execution Analysis

Let's trace the execution of the code step-by-step:

1.  **`fd1 = open(file1, O_WRONLY);`**

      * `file1` is opened with `O_WRONLY`. This flag ensures that if `file1` already exists, its content is **truncated to zero length**. If it doesn't exist, it's created.
      * **Result:** At this point, `file1` becomes empty. The initial content `ONE-TWO-THREE-` is gone.

2.  **`fd2 = open(file2, O_RDONLY);`**

      * `file2` is opened for reading. Its content remains `123456789`.

3.  **`x = read(fd2, s, 20);`**

      * The `read` function attempts to read up to 20 bytes from `file2` into the `char` array `s`.
      * `file2` contains `123456789`. This is 9 characters/bytes.
      * The `read` call will successfully read all 9 bytes.
      * **Result:**
          * The buffer `s` will contain `"123456789"` (followed by whatever was in the remaining parts of `s` or null terminators if `read` added them, but that's not relevant for the next step).
          * The variable `x` will store the number of bytes read, which is `9`.

4.  **`write(fd1, s, x);`**

      * The `write` function takes the file descriptor `fd1`, the buffer `s`, and the number of bytes to write `x` (which is `9`).
      * It writes the first 9 bytes from `s` (which are `"123456789"`) into `file1`.
      * **Result:** `file1` now contains `123456789`.

5.  **`close(fd1);`**

      * `file1` is closed, saving its current content.

6.  **`close(fd2);`**

      * `file2` is closed.

## Final Content of `file1`

After the execution of the code, `file1` will contain:

```
123456789
```

# Ανάλυση Άσκησης C: Κατανόηση Unbuffered I/O

## Εισαγωγή

Αυτό το `README` αναλύει μια τυπική άσκηση προγραμματισμού σε C που εστιάζει στην κατανόηση της συμπεριφοράς του I/O buffering (προσωρινής αποθήκευσης εισόδου/εξόδου), ειδικά όταν το `stdout` (standard output) έχει ρυθμιστεί ως "unbuffered" (χωρίς προσωρινή αποθήκευση).

## Η Άσκηση

### Εκφώνηση

"Μελετήστε το παρακάτω κομμάτι κώδικα. Παρατηρήστε ότι η πολιτική εσωτερικής αποθήκευσης (buffering) για το `stdout` έχει τεθεί μέσω χρήσης της `setvbuf` σε 'not buffered'."

**Ο κώδικας:**

```c
#include <stdio.h> // Συμπεριλάβετε αυτή τη βιβλιοθήκη για printf, fprintf, setvbuf
#include <string.h> // Συμπεριλάβετε αυτή τη βιβλιοθήκη για strlen

int main() { // Προσθήκη main function για να είναι πλήρης ο κώδικας
    setvbuf(stdout, NULL, _IONBF, 0);
    char str[] = "ab\ncd\n";
    int i;

    for (i=0; i < strlen(str); i++) {
        fprintf(stderr, ".");
        printf("%c", str[i]);
    }
    return 0; // Προσθήκη return statement
}
```

**Ποια θα είναι η έξοδος του παραπάνω κώδικα;**

### Επιλογές Απάντησης

  * **A.**
    ```
    -a-b
    -c-d
    ```
  * **B.**
    ```
    ----ab
    cd
    ```
  * **C.**
    ```
    ---ab
    ---cd
    ```
  * **D.**
    ```
    ------ab
    cd
    ```
  * **E.**
    ```
    --ab
    --cd
    ```
  * **F.**✅
    ```
    -a-b-
    -c-d-
    ```
    *(Σημείωση: Στην επιλογή F, ο χαρακτήρας `-` αντιπροσωπεύει την τελεία `.` που τυπώνεται από το `fprintf(stderr, ".")`.)*

-----

## Αναλυτική Απάντηση

Ας αναλύσουμε τον κώδικα βήμα προς βήμα για να κατανοήσουμε πώς θα διαμορφωθεί η έξοδος.

1.  **`setvbuf(stdout, NULL, _IONBF, 0);`**
    Αυτή είναι η πιο κρίσιμη γραμμή. Η συνάρτηση `setvbuf` χρησιμοποιείται για να ορίσει τον τρόπο buffering ενός stream. Εδώ, το `stdout` (το standard output stream) ρυθμίζεται με την επιλογή `_IONBF`, που σημαίνει **"No Buffering"** (χωρίς προσωρινή αποθήκευση). Αυτό σημαίνει ότι κάθε χαρακτήρας που στέλνεται στο `stdout` (μέσω `printf`) θα τυπωθεί στην κονσόλα αμέσως, χωρίς καθυστέρηση.

2.  **`char str[] = "ab\ncd\n";`**
    Δηλώνεται ένας πίνακας χαρακτήρων `str` που περιέχει την ακολουθία `a`, `b`, χαρακτήρας νέας γραμμής (`\n`), `c`, `d`, και πάλι χαρακτήρας νέας γραμμής (`\n`). Ακολουθείται από τον null terminator `\0`, τον οποίο το `strlen` αγνοεί. Το μήκος της συμβολοσειράς `strlen(str)` είναι 6.

3.  **Ο Βρόχος `for`:**
    Ο βρόχος θα επαναληφθεί για κάθε χαρακτήρα της συμβολοσειράς `str`, από τον δείκτη `0` έως τον `5` (συνολικά 6 φορές).

    Μέσα στο βρόχο, για κάθε επανάληψη, εκτελούνται δύο εντολές:

      * **`fprintf(stderr, ".");`**
        Αυτή η εντολή τυπώνει μια **τελεία** (`.`) στο `stderr` (standard error stream). Το `stderr` είναι συνήθως unbuffered από προεπιλογή, οπότε η τελεία θα εμφανιστεί στην κονσόλα αμέσως.

      * **`printf("%c", str[i]);`**
        Αυτή η εντολή τυπώνει τον τρέχοντα χαρακτήρα του `str` (π.χ., `str[0]` είναι `a`, `str[1]` είναι `b` κ.ο.κ.) στο `stdout`. Επειδή το `stdout` έχει ρυθμιστεί ως unbuffered, αυτός ο χαρακτήρας θα εμφανιστεί επίσης αμέσως μετά την τελεία που τυπώθηκε στο `stderr`.

### Βήμα-προς-Βήμα Έξοδος:

Ας δούμε τι θα συμβεί σε κάθε επανάληψη του βρόχου:

  * **`i = 0` (χαρακτήρας 'a'):**

      * `fprintf(stderr, ".")`  -\> τυπώνει `.`
      * `printf("%c", 'a')`   -\> τυπώνει `a`
      * **Τρέχουσα έξοδος στην κονσόλα:** `.a`

  * **`i = 1` (χαρακτήρας 'b'):**

      * `fprintf(stderr, ".")`  -\> τυπώνει `.`
      * `printf("%c", 'b')`   -\> τυπώνει `b`

## Λύση και Επεξήγηση

Η λύση βασίζεται στις θεμελιώδεις ιδιότητες ενός Δυαδικού Δέντρου Αναζήτησης (BST):

  * Όλοι οι κόμβοι στο **αριστερό υποδέντρο** ενός κόμβου `X` έχουν κλειδί **μικρότερο** από το `X->key`.
  * Όλοι οι κόμβοι στο **δεξί υποδέντρο** ενός κόμβου `X` έχουν κλειδί **μεγαλύτερο** από το `X->key`.

Αναλύουμε κάθε επιλογή:

  * **A. `n->left`**:

      * Είναι ο αριστερός απόγονος του `n`. Ως εκ τούτου, `n->left->key < n->key`.
      * **Ισχύει** (`<=` ικανοποιείται).

  * **B. `n->parent`**:

      * Ο γονέας μπορεί να έχει κλειδί μεγαλύτερο ή μικρότερο από το `n->key` (εξαρτάται αν το `n` είναι αριστερό ή δεξί παιδί του γονέα). Δεν υπάρχει εγγύηση.
      * **Δεν Ισχύει**.

  * **C. `n->parent->left`**:

      * **Περίπτωση 1: Το `n` είναι αριστερό παιδί του `n->parent`**. Τότε `n->parent->left` είναι ο ίδιος ο κόμβος `n`. Άρα, `n->parent->left->key == n->key`.
      * **Περίπτωση 2: Το `n` είναι δεξί παιδί του `n->parent`**. Τότε `n->parent->key < n->key`. Επίσης, `n->parent->left->key < n->parent->key`. Άρα, `n->parent->left->key < n->key`.
      * Και στις δύο περιπτώσεις, η συνθήκη `key <= n->key` ικανοποιείται.
      * **Ισχύει**.

  * **D. `n->left->right`**:

      * Ο κόμβος `n->left` έχει κλειδί μικρότερο από `n->key`.
      * Ο κόμβος `n->left->right` βρίσκεται στο δεξιό υποδέντρο του `n->left`, οπότε `n->left->right->key > n->left->key`.
      * Ωστόσο, όλοι οι κόμβοι στο αριστερό υποδέντρο του `n` (συμπεριλαμβανομένου του `n->left->right`) έχουν κλειδί μικρότερο από `n->key`.
      * **Ισχύει** (`<=` ικανοποιείται).

  * **E. `n->right`**:

      * Είναι ο δεξιός απόγονος του `n`. Ως εκ τούτου, `n->right->key > n->key`.
      * **Δεν Ισχύει**.

  * **F. `n->right->left`**:

      * Ο κόμβος `n->right` έχει κλειδί μεγαλύτερο από `n->key`.
      * Ο κόμβος `n->right->left` βρίσκεται στο αριστερό υποδέντρο του `n->right`. Επομένως, `n->right->left->key < n->right->key`.
      * Ωστόσο, το `n->right->left` αντιπροσωπεύει τον *μικρότερο* κόμβο στο δεξιό υποδέντρο του `n` (γνωστός και ως Inorder Successor). Επομένως, το κλειδί του θα είναι πάντα `>= n->key`. Δεν είναι εγγυημένα `<= n->key`.
      * **Δεν Ισχύει**.

  * **G. `n->parent->right`**:

      * **Περίπτωση 1: Το `n` είναι αριστερό παιδί του `n->parent`**. Τότε `n->parent->key > n->key`. Επίσης, `n->parent->right->key > n->parent->key`. Άρα, `n->parent->right->key > n->key`. Δεν ικανοποιείται η συνθήκη.
      * **Περίπτωση 2: Το `n` είναι δεξί παιδί του `n->parent`**. Τότε `n->parent->right` είναι ο ίδιος ο κόμβος `n`. Άρα, `n->parent->right->key == n->key`.
      * Επειδή η Περίπτωση 1 δεν ικανοποιεί τη συνθήκη, δεν υπάρχει εγγύηση.
      * **Δεν Ισχύει**.

# Differences Between Pipes and SOCK_STREAM Sockets

What are the differences between a pipe and a SOCK_STREAM socket?
[Select one or more answers. Incorrect choices are negatively graded.]

---

### Options and Analysis

Here's an analysis of each option:

#### Option A

**Statement:** Η μετάδοση δεδομένων μέσω του αγωγού είναι αξιόπιστη, ενώ μέσω του υποδοχέα όχι.
**Translation:** Data transmission through the pipe is reliable, while through the socket it is not.
**Analysis:** **False.** Both pipes and SOCK_STREAM sockets (which use TCP) provide reliable data transmission. Data sent through both mechanisms is guaranteed to arrive in order and without loss (within system limits for pipes, and over the network for TCP sockets).

#### Option B

**Statement:** Ο αγωγός είναι προσπελάσιμος μέσω ενός περιγραφέα αρχείου, ενώ ο υποδοχέας όχι.
**Translation:** The pipe is accessible via a file descriptor, while the socket is not.
**Analysis:** **False.** Both pipes and sockets are fundamental I/O mechanisms in Unix-like systems and are accessed through file descriptors. This allows them to be used with standard file I/O functions like `read()` and `write()`.

#### Option C

**Statement:** Η επικοινωνία μέσω αγωγού είναι μονόδρομη (half-duplex), ενώ μέσω υποδοχέα αμφίδρομη (full-duplex).
**Translation:** Communication via a pipe is unidirectional (half-duplex), while via a socket it is bidirectional (full-duplex).
**Analysis:** **True.** A standard Unix pipe is a one-way communication channel (data flows from one end to the other). To achieve two-way communication with pipes, you typically need two separate pipes. In contrast, a single SOCK_STREAM socket connection (TCP) is inherently bidirectional, allowing data to be sent and received simultaneously over the same connection.

#### Option D

**Statement:** Η μετάδοση δεδομένων μέσω του αγωγού μπορεί να μπλοκάρει, ενώ μέσω του υποδοχέα όχι.
**Translation:** Data transmission through the pipe can block, while through the socket it cannot.
**Analysis:** **False.** Both pipes and SOCK_STREAM sockets can exhibit blocking behavior by default. A `read()` operation on an empty pipe or socket will block until data is available. Similarly, a `write()` operation can block if the buffer is full. While both can be configured for non-blocking I/O, the general statement that sockets "cannot" block is incorrect for their default behavior.

#### Option E

**Statement:** Ο αγωγός χρησιμοποιείται για την μεταφορά bytes, ενώ ο υποδοχέας για την μεταφορά διακριτών μηνυμάτων.
**Translation:** The pipe is used for byte transfer, while the socket is used for discrete message transfer.
**Analysis:** **False.** Both pipes and SOCK_STREAM sockets are "byte-stream" oriented. They treat data as a continuous stream of bytes and do not preserve message boundaries. If you write three separate blocks of data to a SOCK_STREAM socket, the receiver might read them as a single block or multiple arbitrary-sized blocks. Sockets designed for "discrete messages" are datagram sockets (e.g., SOCK_DGRAM, UDP).

```c
    n->prev = where->prev;
    n->next = where;
    where->prev = n;

```

A. Incorrect amount of memory is allocated, therefore a segmentation fault may occur.

B. The new node is not correctly connected to `where`.

C. The new node is added correctly, but after the function returns, it no longer exists in the list because it was declared locally.

D. The new node is not correctly connected to the one that was before `where` in the original list.

E. If the list is empty, then a segmentation fault will occur.

F. There is no error, the code works correctly in all cases and the list will have the expected form after its execution.

G. If the list contains only one node, then a segmentation fault will occur.

## Answer

The function `addBefore` contains a critical logical error and a syntax error.

**Detailed Analysis:**

1.  **Logical Error (Data Structure Corruption):**

      * When inserting a new node `n` before `where`, the `prev` pointer of `n` is correctly set (`n->prev = where->prev;`), and the `next` pointer of `n` is correctly set (`n->next = where;`). Also, `where->prev` is correctly updated to point to `n`.
      * However, the **crucial missing step** for a doubly linked list is to update the `next` pointer of the node that was *originally before* `where` (i.e., `where->prev`). If `where->prev` is not `NULL` (meaning `where` is not the head of the list), then `where->prev->next` must be updated to point to the new node `n`. The current code fails to do this. This leaves the forward link from the preceding node pointing to `where`, effectively bypassing `n` and breaking the integrity of the list when traversed forwards.

2.  **Syntax Error (Compilation Failure):**

      * The function is declared with `void addBefore(...)`, meaning it should not return any value. However, the last line is `return n;`. This is a syntax error in C and will prevent the code from compiling.

**Evaluation of Options:**

  * **A. False:** `malloc(sizeof(nodeT))` allocates the correct amount of memory.
  * **B. False:** The connections `n->next = where;` and `where->prev = n;` correctly link the new node with `where`.
  * **C. False:** Memory allocated with `malloc` is on the heap and persists beyond the function's scope until explicitly `free()`d.
  * **D. True:** This accurately describes the logical error. The `next` pointer of the node `where->prev` is not updated to point to `n`, breaking the forward link.
  * **E. False:** The problem states `where` is a pointer to an *existing node in the list*. Thus, the list is not empty, and `where` is not `NULL`. If `where` *could* be `NULL`, then `where->prev` would indeed cause a segmentation fault. But under the given premise, this is not a case of error.

  * **G. False:** If `where` is the only node, `where->prev` is `NULL`. The code handles `n->prev = NULL;` correctly and does not cause a segmentation fault.
